## Session Start Protocol

The first thing an agent does in any chat session that touches a project context:

1. **Locate the relevant `ACTIVITY.md`.** `ACTIVITY.md` lives **only at parent projects with submodules** — it's a rollup view of submodule activity, not a per-project diary. Parent-child relationships are declared explicitly in `OVERVIEW.md` frontmatter (each submodule lists `parent: {name}`, each parent lists `submodules: [...]`) — read the project's `OVERVIEW.md` first to resolve which case applies:
   - Working in a submodule (its `OVERVIEW.md` has `parent: X`) → read `projects/X/ACTIVITY.md`.
   - Working in a parent project (its `OVERVIEW.md` has `submodules: [...]`) → read its own `ACTIVITY.md`.
   - Working in a standalone project (neither field set) → skip this step; there is no `ACTIVITY.md` to read.

2. **Find the date of the last session on this project.** Look in `sessions/_INDEX.md`, filter for sessions whose `scope:` matches this project or any of its submodules, and pick the most recent one before today. Note its date.

3. **Surface what's happened since.** From the `ACTIVITY.md`, list every entry dated **after** the last session's date. Format:

   ```
   Activity in {parent-project} since last session ({YYYY-MM-DD}):
   - YYYY-MM-DD [submodule-name] [issue] Summary → [[submodule-name/ISSUES#issue-007]]
   - YYYY-MM-DD [submodule-name → affects: sibling-name] [decision] Summary → [[submodule-name/OVERVIEW]]
   ```

   If there's no prior session for this project, fall back to the last 5 entries or the last 7 days, whichever is more.

4. **Skip the recap only if you explicitly open with a quick one-off question** ("just answer this quick thing") or name a non-project context (`general`, an idea, a brand file, research topic).

This protocol ensures no agent works blind to what's happened across submodules between sessions.

## Sessions

Chat sessions worth carrying forward — discussions that produced artifacts, decisions, or references useful for later work — are captured under `sessions/{YYYY-MM-DD-slug}/`. Each session is its own folder:

- `SESSION.md` — summary, key points, decisions, linked artifacts and references

Sessions are **flexibly scoped** via a `scope:` field pointing to a vault path: `projects/X`, `ideas/X`, `brand/X`, `research/topics/X`, or `general`. Not every session ties to a project.

Starter defaults — edit freely to match how you actually work. These apply

## Write Protocol

When you discover something worth persisting — an issue, a fix, a decision, a progress update, an idea — **propose the write inline**, wait for approval, then write directly to the destination file. No pending-review log; no staging. Inline review IS the review.

### Process

1. Identify the primary destination from the entry type and project context (see table below).
2. Resolve cross-module impact (see [Cross-Module Awareness](#cross-module-awareness)).
3. Show the full draft of the insert/edit, not a summary. If multiple files will be touched (primary + breadcrumbs), show all of them in one proposal.
4. Wait for explicit approval — "yes", "approve", "write it", or specific edits.
5. Write all destinations in one batched pass.
6. Confirm in chat exactly what was written.

### Destination resolution

| Type | Primary destination |
|------|---------------------|
| `issue` | `projects/{project}/ISSUES.md` |
| `resolution` | Update the matching issue in `projects/{project}/ISSUES.md` |
| `progress` | `projects/{project}/PROGRESS.md` |
| `decision` | `projects/{project}/OVERVIEW.md` (Decisions section) |
| `context` | `projects/{project}/OVERVIEW.md` (Notes section) |
| `idea` | `ideas/{domain}.md` |

For non-project entries: brand/research updates go to their respective files (`brand/PROFILE.md`, `research/topics/{topic}.md`, etc.) using the same propose-then-write flow.

### Entry format

Since entries write directly to destination files, the destination's existing format governs. Follow whatever convention the file uses — issue blocks in `ISSUES.md`, milestone rows in `PROGRESS.md`, idea blocks in `ideas/{domain}.md`, source/topic structure in research. Don't invent new frontmatter for entries inside existing files.

### Rules

- **Never write without explicit approval.** Even small edits.
- **Show the full draft, not a summary.**
- **Batch related writes.** Primary + breadcrumbs go up as one proposal, get one approval, write together.
- **Be specific.** Reference file paths, line numbers, commit SHAs, URLs.
- **Don't invent details.** Only write what was actually discussed.
- **Resolutions reference the issue ID** they resolve (e.g. "Resolves ISSUE-001").

## Cross-Module Awareness

When a project has submodules (other project folders nested inside it), the parent must always know what happens in any submodule. The parent maintains an `ACTIVITY.md` that aggregates submodule activity as chronological breadcrumbs. **Standalone projects (no submodules) do not have an `ACTIVITY.md`** — there is nothing to roll up.

**Parent-child relationships are declared in `OVERVIEW.md` frontmatter.** Each submodule lists `parent: {parent-name}`; each parent lists `submodules: [child1, child2, ...]`. This is the source of truth for resolving which `ACTIVITY.md` to read at session start and where to drop breadcrumbs. Folder nesting is incidental; the frontmatter declaration is authoritative.

### Rules

1. **`ACTIVITY.md` lives at parent projects only.** Located at `projects/{parent}/ACTIVITY.md`. Submodules do not have their own — they keep their work in their own `ISSUES.md` / `PROGRESS.md` / `OVERVIEW.md` as usual. All cross-module signaling routes through the parent's single feed.

2. **Parent breadcrumb is implicit.** Every substantive submodule write generates a breadcrumb in the parent's `ACTIVITY.md`. The agent doesn't request this — it's part of every submodule proposal automatically.

3. **Sibling breadcrumbs are explicit, but go in the parent's feed.** Use `affects: [sibling1, sibling2]` in the proposal to declare which sibling submodules are impacted. The breadcrumb in the parent's `ACTIVITY.md` carries the `→ affects: ...` marker so anyone reading the parent (including agents working in the affected sibling at session start) sees the cross-impact.

4. **`affects:` accepts sibling submodule names only** (not full vault scopes). E.g. `affects: [module-a, module-b]`. Cross-project impact is rare; if it genuinely happens, propose a separate entry under the other project.

5. **Decision heuristic** — when proposing a submodule write, ask:
   - Does this change a shared contract (protocol, schema, public API)? → list every consumer in `affects:`
   - Does this change observable behaviour another module depends on? → list those modules
   - Is this purely internal? → leave `affects:` empty; parent still gets the breadcrumb implicitly

6. **Which entry types trigger breadcrumbs?**

   | Type | Parent breadcrumb |
   |------|-------------------|
   | `issue` | ✓ |
   | `resolution` | ✓ |
   | `progress` | ✓ |
   | `decision` | ✓ |
   | `context` | ✗ (too noisy) |
   | `idea` | n/a (ideas are domain-scoped) |

### Breadcrumb format

Single line, chronological append (most recent at top of the most-recent date block) in the parent's `ACTIVITY.md`:

**Submodule-only change:**

```
- YYYY-MM-DD [submodule-name] [issue] Long-poll timeout dropped from 30s to 15s. → [[submodule-name/ISSUES#issue-007]]
```

**Submodule change that affects siblings (`affects:` is set):**

```
- YYYY-MM-DD [submodule-name → affects: sibling-a, sibling-b] [issue] Long-poll timeout dropped from 30s to 15s. Sibling-a must retune. → [[submodule-name/ISSUES#issue-007]]
```

One entry per write, even when multiple siblings are affected — the `→ affects:` marker lists all of them.

### ACTIVITY.md skeleton (parent projects only)

If the parent doesn't have one yet, create it as part of the first activity write. Skeleton:

````markdown
---
type: activity
project: {parent-project-name}
last_updated: YYYY-MM-DD
---

# {Parent Project} — Activity Feed

> Rollup of submodule activity. Each submodule's substantive work
> generates a breadcrumb here so the parent stays aware of all moving
> parts. Most recent at top.

---

## YYYY-MM-DD

- YYYY-MM-DD [submodule-name] [type] Summary. → [[submodule-name/ISSUES#...]]
````

One date heading per day; entries within a day in any order.
